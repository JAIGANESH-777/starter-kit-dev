# Accelerator Scaffolder Skill

Use this skill when initializing, building, or modifying projects generated by the Accelerator Enterprise Compiler. This skill teaches the agent how to read target configurations from `SPEC.md` and translate them into a fully operational, multi-container environment.

## 1. Stack Discovery Phase
Before writing any code, examine the current configurations:
- Read `SPEC.md` to identify:
  - **Operational Topology**: fullstack, backend, or frontend.
  - **Frontend Core**: Next.js, React+Vite, Nuxt 3, or SvelteKit.
  - **Backend Core**: NestJS, FastAPI, Django, Express, or Hono.
  - **Database & ORM**: PostgreSQL or SQLite; Prisma, Drizzle, Django ORM, or SQLAlchemy.
  - **Authentication**: JWT Stateless, SSO, or Hybrid (using Auth.js, Passport, or Custom).
  - **Cloud Platform**: AWS, Azure, GCP, or Docker-Compose.

## 2. Bootstrapping Sequence (Strict Pipeline)

### Step 1: Root Foundations & Variables
- Define `.env.example` configurations in the backend and frontend roots.
- Ensure backend vars use variables like `JWT_SECRET` or `BACKEND_SECRET` (unless Django is active, which uses `SECRET_KEY`).
- Ensure frontend connections point to `http://localhost:8000/api` using framework variables like `NEXT_PUBLIC_API_URL`, `VITE_API_URL`, or `PUBLIC_API_URL`.

### Step 2: Database Initialization & Calibration
- If **Prisma** is selected, verify the connection protocol in the datasource block is `postgresql://` (not `postgres://`). Write wrapper scripts or runtime overrides if the global env provides `postgres://` (C-2 Connection Fix).
- If **SQLite** is selected, write local DB files to `/app/db/local.db` and bypass global PostgreSQL servers.
- Generate migrations and seeding scripts.

### Step 3: Core Backend Verification (L-2 Rule)
- Scaffold the backend application logic and create a functional `/health` controller endpoint.
- **Verification checkpoint**: You must compile, build, and verify the backend `/health` endpoint returns HTTP 200 (using mocked services where necessary) before writing any frontend UI component!

### Step 4: Frontend Scaffolding & Styling
- Scaffold the frontend framework in the `frontend` directory using non-interactive CLI flags.
- **Design Tokens Integration (W-4 Fix)**: Read colors and spacing from `shared_tokens.md`. Convert HEX values into native CSS HSL fragments to allow correct color theme mappings.
- Implement a visible Dark/Light theme toggle using `data-theme` selectors.
- Create dashboard cards presenting system stats and an active connection query to the backend health endpoint.

### Step 5: Docker & Cloud Orchestration
- Construct production-ready multi-stage Dockerfiles:
  - Frontend: must `EXPOSE 3000`.
  - Backend: must `EXPOSE 8000`.
- Generate Terraform deployment templates matching the chosen target cloud (AWS, Azure, GCP).
- Generate the root `docker-compose.yml` by copying and applying the dynamically compiled Docker service configurations from Section 6 of `SPEC.md`. Ensure all port mappings, dependent services (db, redis, keycloak), and healthchecks match the specification exactly.
