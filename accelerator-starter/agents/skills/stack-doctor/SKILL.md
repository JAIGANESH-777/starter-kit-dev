> **Scope boundary**: If the project hasn't been scaffolded yet, hand off to the `accelerator-scaffolder` skill. If the user is writing new features or tests, this skill does not apply.

## 2. Context Gathering (Do This First)

Before diagnosing, read the project's own configuration â€” do NOT rely on hardcoded defaults:

1. **Read `SPEC.md`** â€” identify the active frontend, backend, database, ORM, auth, and cloud platform. All diagnostic steps below adapt to whatever stack SPEC.md declares.
2. **Read `docker-compose.yml`** â€” note the declared service names, port mappings, and environment variable references.
3. **Read `.env` / `.env.example`** files in `backend/` and `frontend/` â€” note what variables are set and what values they hold.
4. **Check recent build output or error logs** the user has shared.

> Do not re-teach scaffolding rules (port conventions, env var naming, connection string protocols). Those are defined in `SPEC.md` sections C-1 and C-2. Reference them; don't duplicate them.

## 3. Diagnostic Playbooks

### 3.1 Docker Daemon Issues

**Symptom**: `Cannot connect to the Docker daemon` or Docker commands hang.

```
â†’ Is Docker Desktop installed and running?
  â”śâ”€ NO â†’ Start Docker Desktop, wait for it to initialize
  â””â”€ YES â†’ Restart Docker Desktop; if on Windows, check WSL2 backend is enabled
```

### 3.2 Build-Time Failures

**Symptom**: `docker compose up --build` fails during the `docker build` step.

| Check | What to look for |
|:---|:---|
| Dockerfile syntax | Verify the failing Dockerfile exists and has valid `FROM`, `COPY`, `RUN` stages |
| Package install failure | `npm install` or `pip install` failing â†’ delete lockfile + `node_modules`, rebuild |
| Missing files in COPY | A `COPY` references a file that doesn't exist â†’ check `.dockerignore` isn't excluding it |
| Wrong base image | Base image doesn't match the framework's runtime (e.g., Python image for a Node backend) |

### 3.3 Container Startup Crashes

**Symptom**: A container starts then exits with code 1. `docker compose ps` shows it as `exited`.

**Diagnostic steps:**
1. Run `docker compose logs <service-name>` â€” read the last 30 lines for the actual error.
2. Common causes:
   - **Missing environment variable** â€” the app tries to read a required var that isn't set. Cross-check `.env` against what the framework expects (refer to SPEC.md Section 6 for env var naming rules).
   - **Database not ready** â€” the backend starts before the DB accepts connections. Add a health check or `depends_on` with `condition: service_healthy` in `docker-compose.yml`.
   - **Port already in use inside container** â€” unlikely in Docker, but check if `EXPOSE` port matches the app's listen port.

### 3.4 Port / Networking Failures

**Symptom**: `port is already allocated` or `ECONNREFUSED` between containers.

| Symptom | Diagnosis | Fix |
|:---|:---|:---|
| `port is already allocated` | Another process holds that port on the host | Find it: `netstat -ano \| findstr :<PORT>`, then kill the PID |
| `ECONNREFUSED` from frontendâ†’backend | Frontend uses `localhost` instead of the Docker service name | Inside Docker network, use the service name from `docker-compose.yml` (e.g., `backend`, not `localhost`) |
| `no such service` in compose | Service name typo | Check `docker-compose.yml` service names exactly match references in env vars |

### 3.5 Backend Runtime Errors

**Symptom**: Backend container is running but returns errors or doesn't respond.

Read SPEC.md to identify the active backend framework, then apply the matching playbook:

#### NestJS
| Symptom | Fix |
|:---|:---|
| `Nest can't resolve dependencies of X` | Check `@Injectable()` decorator and module `providers` array |
| `Unknown element` in module | Import the module in the consuming module's `imports` array |
| Circular dependency warning | Use `forwardRef(() => ModuleName)` |
| Guard not applied | Ensure `@UseGuards()` is on the controller/route |

#### Express / Hono
| Symptom | Fix |
|:---|:---|
| `Cannot GET /api/health` | Route not registered â€” check route file is imported and mounted |
| Middleware ordering issues | Error handlers must be registered last, after all routes |

#### FastAPI
| Symptom | Fix |
|:---|:---|
| `422 Unprocessable Entity` | Request body doesn't match Pydantic model; check field types and required fields |
| `ModuleNotFoundError` | Missing dependency; check `requirements.txt` or `pyproject.toml` |
| CORS errors from browser | Add `CORSMiddleware` with allowed origins matching the frontend URL |

#### Django
| Symptom | Fix |
|:---|:---|
| `ImproperlyConfigured` | Check `INSTALLED_APPS` and `settings.py` configuration |
| Static files 404 in production | Run `python manage.py collectstatic` |
| `DisallowedHost` | Add the hostname to `ALLOWED_HOSTS` in settings |

### 3.6 Cross-Stack Wiring Failures

**Symptom**: Frontend loads but API calls fail with network errors or CORS issues.

**Diagnostic steps:**
1. **Identify the frontend env var** â€” read SPEC.md to find which variable the frontend uses for the API URL (e.g., `NEXT_PUBLIC_API_URL`, `VITE_API_URL`, `PUBLIC_API_URL`, `NUXT_PUBLIC_API_URL`).
2. **Check the value**:
   - For **browser-side requests** (client components): must use `http://localhost:<backend-port>/api`
   - For **server-side requests** (SSR in Next.js, Nuxt, SvelteKit): must use `http://<backend-service-name>:<backend-port>/api` inside Docker
3. **Verify CORS** â€” the backend must allow the frontend's origin. Check middleware configuration.
4. **Verify the backend is reachable** â€” from inside the frontend container: `curl http://<backend-service>:<port>/api/health`

> SSR frameworks have a split-brain problem: server-side fetches run inside the Docker network (use service name), client-side fetches run in the browser (use `localhost`). Both paths must work.

### 3.7 Database / Migration Issues

**Symptom**: App connects but queries fail, or migrations won't run.

**Diagnostic steps:**
1. **Check connection string protocol** â€” refer to SPEC.md C-2 rule for the active ORM's required format.
2. **Check host resolution** â€” inside Docker, the DB host is the service name (e.g., `db`), not `localhost`.
3. **Check credentials** â€” `POSTGRES_PASSWORD` in `docker-compose.yml` must match the password in `DATABASE_URL`.

| Symptom | ORM | Fix |
|:---|:---|:---|
| `P1001: Can't reach database server` | Prisma | DB host wrong â€” use `db` in Docker, `localhost` outside |
| `error: password authentication failed` | Any | Credentials mismatch between compose env and connection string |
| `relation does not exist` / `no such table` | Prisma/Drizzle | Run migrations: `npx prisma migrate dev` or `npx drizzle-kit push` |
| `database "X" does not exist` | Any | `POSTGRES_DB` in compose doesn't match DB name in connection string |
| `SQLITE_CANTOPEN` | Prisma/Drizzle | Target directory doesn't exist or has wrong permissions |
| `OperationalError: no such table` | Django/SQLAlchemy | Run `python manage.py migrate` or `alembic upgrade head` |

## 4. Post-Fix Verification

After applying a fix, validate the full stack in this order. Stop at the first failure:

```
1. [DB]      docker compose up db -d â†’ wait for healthy
2. [MIGRATE] Run ORM migrations against the database
3. [BACKEND] Build + start backend â†’ verify /health returns HTTP 200
4. [FRONTEND] Build + start frontend â†’ verify page loads
5. [DOCKER]  docker compose up --build â†’ all containers healthy
6. [WIRING]  Frontend can call backend API and display data
```

Health check commands (adapt port/path to your SPEC.md):
```bash
# Backend health (typical paths per framework)
curl http://localhost:8000/api/health    # NestJS / Express / Hono
curl http://localhost:8000/health        # FastAPI
curl http://localhost:8000/api/health/   # Django

# Frontend
curl http://localhost:3000
```

## 5. Dependency Conflicts

Only investigate dependencies **after** the triage tree above has ruled out configuration issues.

```bash
# Node.js â€” check for outdated or conflicting packages
npx npm-check-updates
npm ls <package-name>    # trace dependency tree for a specific conflict

# Python
pip list --outdated
```

| Common conflict | Resolution |
|:---|:---|
| `@prisma/client` version â‰  `prisma` CLI | Both must be the same version in `package.json` |
| `react` version vs `next` version | Next.js pins React; don't override manually |
| `@nestjs/*` at mixed major versions | All `@nestjs` packages must share the same major |
| Peer dependency warnings | Resolve them â€” they cause silent runtime failures. Use `npm ls` to trace, `overrides` as last resort |
